## BUG-01 · 2026-05-xx · Recharts Tooltip 类型错误

**问题描述**  
`app/admin/DashboardClient.tsx` 构建报错：  
```
Argument of type '(value: number) => [number, string]' is not assignable to parameter of type 'Formatter<...>'
```

**根本原因**  
Recharts `<Tooltip>` 的 `formatter` 函数参数类型签名与实际接收类型不匹配，  
手动标注 `: number` 导致 TypeScript 报错。

**解决方式**  
移除参数的手动类型注解，改为让 TypeScript 推断：  
```tsx
// 修改前
formatter={(value: number) => [value, '访问量']}
// 修改后
formatter={(value) => [value, '访问量']}
```

## BUG-03 · 2026-05-xx · 容器 restart loop（四个子原因）

**问题描述**  
Docker 容器部署后持续重启，日志反复出现 Prisma 相关错误。

### 子原因 A：schema.prisma 无 url 字段

**根本原因**  
Prisma 7 改用 `prisma.config.ts` 配置 datasource URL，  
`schema.prisma` 的 `datasource db {}` 块**不再支持** `url` 字段。  
曾尝试添加 `url = env("DATABASE_URL")` 导致报错：  
`"datasource property url is no longer supported in schema files"`

**解决方式**  
`schema.prisma` 的 datasource 块不写 url：  
```prisma
datasource db {
  provider = "sqlite"
  # 无 url 字段
}
```  
URL 统一在 `prisma.config.ts` 的 `datasource.url` 中配置。

### 子原因 B：prisma.config.ts 未复制进 runner 镜像

**根本原因**  
Dockerfile runner 阶段缺少 `COPY --from=builder /app/prisma.config.ts ./`，  
`npx prisma db push` 在容器内找不到配置，报 config 错误。

**解决方式**  
在 Dockerfile runner 阶段补充：  
```dockerfile
COPY --from=builder /app/prisma.config.ts ./
```

### 子原因 C：prisma.config.ts 导入了 devDependency

**根本原因**  
`prisma.config.ts` 有 `import "dotenv/config"`，  
但 `dotenv` 是 devDependency，生产镜像没有安装，导致 import 失败。

**解决方式**  
删除 `import "dotenv/config"` 一行，  
DATABASE_URL 由 docker-compose `environment:` 直接注入。

### 子原因 D：lib/db/client.ts fallback 路径写死 Docker 路径

**根本原因**  
```typescript
const dbUrl = process.env.DATABASE_URL ?? path.resolve(process.cwd(), 'dev.db')
```  
本地开发时 `cwd()` 能找到 `dev.db`，但容器内路径不符实际挂载点。

**解决方式**  
改为统一的绝对路径字符串：  
```typescript
const dbUrl = process.env.DATABASE_URL ?? 'file:/app/data/dev.db'
```

## BUG-04 · 2026-05-xx · Live2D SSR 崩溃

**问题描述**  
生产构建时服务端渲染阶段报错：  
`ReferenceError: document is not defined`  
`Error: ssr: false is not allowed for Server Components`

**根本原因**  
`l2d-widget` 在模块加载时立即执行 `document.head.append(...)` 等浏览器专属 API；  
且 `next/dynamic({ ssr: false })` 不能在 Server Component 中直接使用。

连续踩了三个坑：
1. `import dynamic from 'next/dynamic'` 与文件顶层 `export const dynamic` 命名冲突
2. Turbopack 不允许 import 语句之间插 `const` 声明
3. `ssr: false` 只能用于 Client Component

**解决方式**  
新建 `components/live2d/Live2DWidgetLoader.tsx` 作为 Client Component 边界：  
```tsx
'use client'
import dynamic from 'next/dynamic'

const Live2DWidget = dynamic(
  () => import('./Live2DWidget').then(m => m.Live2DWidget),
  { ssr: false }
)
export function Live2DWidgetLoader({ ... }) {
  return <Live2DWidget ... />
}
```  
`layout.tsx`（Server Component）只引用 `<Live2DWidgetLoader />`。

## BUG-05 · 2026-05-25 · 后台登录 ADMIN_PASSWORD 环境变量无效

**问题描述**  
VPS 上通过 `.env.local` 注入 `ADMIN_PASSWORD=Ashia2024!`，  
但后台 `/admin/login` 用该密码登录失败（返回 401）。

**根本原因**  
`app/api/auth/route.ts` 的 `verifyPassword` 函数在 DB 无存储密码时，  
直接与内置 `DEFAULT_PASSWORD_HASH`（admin123 的哈希）比较，  
完全跳过了 `ADMIN_PASSWORD` 环境变量：  
```typescript
// 旧代码
if (stored) return hashPassword(input) === stored
return hashPassword(input) === DEFAULT_PASSWORD_HASH  // 跳过了 env
```

**解决方式**  
在默认密码之前插入环境变量检查：  
```typescript
if (stored) return hashPassword(input) === stored
const envPassword = process.env.ADMIN_PASSWORD
if (envPassword) return hashPassword(input) === hashPassword(envPassword)
return hashPassword(input) === DEFAULT_PASSWORD_HASH
```  
验证：本地直接查 DB 确认无 `adminPasswordHash` 记录 → 环境变量路径生效。

## BUG-06 · 2026-05-25 · Live2D canvas 被移走导致渲染断链

**问题描述**  
Live2D 角色完全不显示（无任何报错，canvas 空白）。

**根本原因**  
`l2d-widget` 的 DOM 结构为：  
```
body
└── div.container (position:fixed; bottom:0; left:0; z-index:9999)
    └── canvas   ← l2d 内部渲染循环持有此引用
```

旧代码 900ms 后执行：  
```typescript
wrapperRef.current.appendChild(canvas)   // 把 canvas 从 l2d-widget 的 container 里移走！
```  
这切断了 l2d 内部渲染循环与 canvas 的绑定，canvas 永久空白。

**解决方式**  
不再移动 canvas，改为引用 `canvas.parentElement`（即 l2d-widget 的 container div），  
拖拽 transform 直接写到该 container 的 style：  
```typescript
const container = canvas?.parentElement as HTMLElement
widgetContainerRef.current = container
container.style.transform = `translate(${x}px, ${y}px)`
```  
同时将 React 覆盖层 z-index 从 `z-35/36/40` 改为 `9998/10000/10001`，  
确保真正叠在 l2d-widget 的 canvas(9999) 之上。

## BUG-07 · 2026-05-26 · Live2D 仍不显示（三个深层根因）

**问题描述**  
BUG-06 修复后 Live2D 依然不显示。

### 根因 A：双重初始化竞态

**根本原因**  
Zustand store 默认值为 `live2dModel='blanc'`, `live2dSize=220`，  
`StoreHydrator` 首次 useEffect 将其更新为 DB 值（如 `golden`, `200`）。  
这导致 `Live2DWidget` 的 `useEffect([live2dModel, live2dSize])` 连续触发两次：  
- 第一次：以 blanc/220 初始化，创建 widget1  
- 第二次：cleanup 销毁 widget1，以 golden/200 初始化，创建 widget2

两个 widget 短暂共存于 DOM。

**解决方式**  
从 `layout.tsx`（服务端已读 DB）直接将 `initialModel` / `initialSize` 作为 props 传给 `Live2DWidget`。  
首次渲染直接使用 DB 值，避免第一次以错误默认值初始化。

### 根因 B：destroy() 永久挂死

**根本原因**  
`l2d-widget` 默认 `transitionType: 'slide'`，  
初始状态为 `transform: translateY(130%)`（屏幕外）。  
在模型加载完成前（`Ko(d, true)` 未执行）就调用 `destroy()`：  
- `Ko(d, false)` 将 transform 设为 `translateY(130%)` → 与初始值相同，无实际过渡  
- `await k()` 监听 `transitionend` → **永远不触发**  
- `destroy()` 挂死，旧 widget 永不从 DOM 移除

结果：两个 widget 永久共存，空白旧 canvas 覆盖新 widget。

**解决方式**  
改用 `transitionType: 'fade'`：  
- `destroy()` 内部走 `setTimeout(transitionDuration)` 而非 `transitionend`  
- 无论是否有视觉过渡，都能在固定时间内完成清理

### 根因 C：querySelector 选中错误 canvas

**根本原因**  
`document.querySelector('canvas')` 返回 DOM 中第一个 canvas，  
在双重初始化期间，选中的是旧 widget1 的 canvas，  
`widgetContainerRef` 指向即将被销毁的 DOM 元素，  
之后所有 transform 写入均无效，拖拽失效。

**解决方式**  
在 `createWidget()` 调用**之前**注册 `MutationObserver` 监听 `body.childList`，  
精确捕获本次 `createWidget` 新增的 container div：  
```typescript
const mutObs = new MutationObserver((mutations) => {
  for (const mutation of mutations) {
    for (const node of mutation.addedNodes) {
      if (node instanceof HTMLElement && node.querySelector('canvas')) {
        widgetContainerRef.current = node
        mutObs.disconnect()
      }
    }
  }
})
mutObs.observe(document.body, { childList: true })
widgetRef.current = createWidget(...)   // MutationObserver 会捕获这次新增的 div
```

## 依赖关系图（Live2D 相关）

```
layout.tsx (Server Component)
└── Live2DWidgetLoader (Client Component, 'use client')
    └── Live2DWidget (动态加载, ssr:false)
        ├── useAppStore (live2dModel, live2dSize)  ← StoreHydrator 从 DB 注入
        ├── import('l2d-widget') → createWidget()
        │   └── DOM: body > div.container(z:9999) > canvas
        │       ↑ MutationObserver 捕获此 div 用于拖拽 transform
        └── React 覆盖层 (z:9998/10000/10001)
            ├── 气泡: z-10000, pointer-events:none
            ├── 投影: z-9998
            └── 拖拽遮罩: z-10001 (捕获 pointer events)
```

```
l2d-widget 生命周期
createWidget()
  → 立即创建 container div (opacity:0, transition:fade 600ms)
  → 加载模型 m.load(path)
  → 模型加载完成 → Ko(d, false, true) → opacity:1 → 角色出现
destroy()
  → Ko(d, false, false) → opacity:0
  → setTimeout(600ms) → m.destroy() → body.removeChild(d)
```
